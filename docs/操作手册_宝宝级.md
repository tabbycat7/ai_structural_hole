# 操作手册：从空跑到正式实验

这份手册面向第一次运行项目的人。请按顺序操作，不要一上来就跑大样本。

本文只负责“怎样运行”。实验原理请看[实验说明_详细版.md](实验说明_详细版.md)，正式跑数前的验收要求请看[validity_threats.md](validity_threats.md)。

## 1. 先分清两种运行

| 运行方式 | 用途 | 能否作为研究结果 |
| --- | --- | --- |
| `--mock` + 模板文章 | 检查安装、命令和输出流程 | 不能 |
| 真实模型 + 冻结题库/基线/变体 | 正式实验 | 通过质检后才可以 |

模板路线和 MockClient 都是工程测试工具。它们能证明程序跑通，不能证明真实模型存在某种偏好。

## 2. 打开终端并进入项目

Windows 打开 PowerShell，macOS/Linux 打开 Terminal。然后进入项目目录：

```powershell
cd "你的项目路径\ai_structural_hole"
```

确认当前目录下能看到 `README.md`、`requirements.txt` 和 `ai_structural_holes` 文件夹。

## 3. 安装依赖

建议使用 Python 3.9 或更高版本：

```powershell
python --version
python -m pip install -r requirements.txt
```

安装完成后运行：

```powershell
python -m ai_structural_holes.cli graph
python -m pytest -q
```

第一条应打印因果图和混杂集；第二条应通过测试。如果系统只有 `python3`，把后续命令中的 `python` 改成 `python3`。

## 4. 用 Mock 做最小空跑

```powershell
python -m ai_structural_holes.cli study1 --mock --per-domain 1 --seeds 1
```

成功后会生成（每个被测模型一个子目录，目录名由模型 slug 转换，如 `deepseek__deepseek-chat`）：

```text
outputs/study1/<model>/trials.csv
outputs/study1/<model>/ate.csv
outputs/study1/<model>/ei_leverage.csv
outputs/study1/<model>/ate_forest.png
outputs/study1/<model>/ei_leverage.png
```

这一阶段只检查流程，不解释数值。

## 5. 配置真实模型密钥

复制环境变量模板：

```powershell
Copy-Item .env.example .env
```

macOS/Linux 使用：

```bash
cp .env.example .env
```

打开 `.env`，按使用的模型填写密钥：

```text
OPENROUTER_API_KEY=你的OpenRouter密钥
DEEPSEEK_API_KEY=你的DeepSeek官方API密钥
QWEN_API_KEY=你的阿里云MaaS密钥
DOUBAO_API_KEY=你的火山引擎Ark密钥
MOONSHOT_API_KEY=你的Moonshot（Kimi）密钥
MINIMAX_API_KEY=你的MiniMax密钥
```

当前路由规则是：

- `deepseek/...` 模型走 DeepSeek 官方 API，需要 `DEEPSEEK_API_KEY`。
- `qwen/...` 或 `qwen*`（如 `qwen3.6-flash`）走阿里云 MaaS 兼容端点，需要 `QWEN_API_KEY`（也支持 `DASHSCOPE_API_KEY`）。
- `doubao/...` 或 `doubao*`（如 `doubao-seed-2-0-mini-260428`）走火山引擎 Ark，需要 `DOUBAO_API_KEY`（也支持 `ARK_API_KEY`）。
- `kimi/...`、`kimi*` 或 `moonshot/...`、`moonshot-*`（如 `kimi-k2`、`moonshot-v1-8k`）走 Moonshot 官方 API，需要 `MOONSHOT_API_KEY`（也支持 `KIMI_API_KEY`）。对 `kimi-k2.5`、`kimi-k2.6`、`kimi-k2` 等思考模型，客户端默认传 `thinking.type=disabled` 关闭思考（可用 `.env` 设 `KIMI_DISABLE_THINKING=0` 恢复），且**不传 `temperature`**（这些模型只接受默认值）；`kimi-k2.7-code` 无法关闭思考。
- `minimax/...`、`minimax*` 或 `abab*`（如 `MiniMax-M3`、`abab6.5s-chat`）走 MiniMax 官方 API，需要 `MINIMAX_API_KEY`。对 `MiniMax-M3` 等思考模型，客户端默认传 `thinking.type=disabled` 关闭思考（可用 `.env` 设 `MINIMAX_DISABLE_THINKING=0` 恢复），否则 `minimal` 模式下 32 token 预算会被思考内容占满、导致 `parse_ok≈0`。
- 其他模型走 OpenRouter，需要 `OPENROUTER_API_KEY`。

没有任何密钥时，客户端可能回退 Mock。正式运行前一定要查看终端警告和对应 API 平台的用量记录。`trials.csv` 中的 `model` 只记录请求时填写的模型名，不能单独证明请求真的发给了该模型。

`.env` 包含密钥，不要提交到 Git，也不要发给其他人。

## 6. 用真实模型做小规模连通测试

如果使用 DeepSeek：

```powershell
python -m ai_structural_holes.cli study1 --models deepseek/deepseek-chat --per-domain 1 --seeds 1
```

如果使用 OpenRouter 模型：

```powershell
python -m ai_structural_holes.cli study1 --models openai/gpt-4o --per-domain 1 --seeds 1
```

这一步仍然使用默认模板材料，只用于确认密钥和网络调用正常。

## 7. 正式实验前准备真实题库

正式实验建议使用真实问题和真实网页段落，而不是内置的少量主题。

如果 `data/query_pool/` 已经有经过确认的冻结题库，可以跳过导入。否则，准备 DuReader 风格的 JSON/JSONL 文件，然后运行：

```powershell
python -m ai_structural_holes.cli import-queries --file search.dev.json --per-domain 50
```

程序会：

1. 清理问题和网页段落；
2. 按关键词分到消费、健康、金融、学术和旅行五个领域；
3. 每个领域最多抽取指定数量；
4. 写入 `data/query_pool/`。

导入后必须人工抽检：

- 问题是否属于标注领域；
- 网页段落是否与问题相关；
- 是否存在乱码、广告、截断或明显错误；
- 每个问题是否有足够的可用段落。

确认后冻结该目录。正式实验期间不要一边运行一边修改题库。

## 8. 生成并冻结基线文章

基线文章是所有处理文和对照文的共同底稿。题库来源和题目数量必须与之后的实验一致或更大。

```powershell
python -m ai_structural_holes.cli gen-base `
  --model deepseek/deepseek-chat `
  --query-source pool `
  --per-domain 50 `
  --concurrency 4
```

以上是 PowerShell 多行写法；也可以写成一行：

```powershell
python -m ai_structural_holes.cli gen-base --model deepseek/deepseek-chat --query-source pool --per-domain 50 --concurrency 4
```

结果存入 `data/base_articles/`。重点检查：

- `generator` 应为 `llm`；
- `template_fallback` 应重新生成或剔除；
- 正文应真正回答或解释问题，而不是泛泛复述题目；
- 不应包含明显事实错误；
- 同一批文章的长度和质量不应极端失衡。

修改生成设置后想重做已有基线，可使用 `--force`。这会改变后续变体的基线哈希，因此正式跑数前要重新冻结变体。

## 9. 预生成 Study 1 变体

建议先单独生成变体并抽检，再启动昂贵的选择实验：

```powershell
python -m ai_structural_holes.cli gen-variants `
  --model deepseek/deepseek-chat `
  --query-source pool `
  --per-domain 50 `
  --concurrency 4
```

必须注意两点：

1. `--query-source pool` 不能漏掉，否则会为内置题目生成变体。
2. 这里的 `--model` 必须与 Study 1 的 `--gen-model` 一致，才能复用冻结变体。

变体写入 `data/variant_articles/`，清单写入：

```text
outputs/study1/variants_manifest.csv
```

当前变体生成只把目标档位作为写作指引，非空正文会直接接受；程序不会自动证明实际正文实现了目标档位。因此正式实验前必须检查正文，而不能只看 `intended_profile`。

建议按“问题 × 特征”抽样检查处理—对照对：

- 是否真正改变了目标特征；
- 其他七个特征是否保持不变；
- 事实、立场和核心答案是否一致；
- 长度是否近似；
- 是否出现模板回退；
- 是否存在生成模型特有的风格或套话。

`gen-variants` 当前主要服务 Study 1。Study 2 的组合会在运行时生成并自动存入变体库。Study 4 请用 **`study4-materials`** 单独生成 S1/S3 的 none/genuine/fake 材料（见第 11 节）。

若扫描发现大量**截断/过短**变体（例如不足 50 字、仅为基线文章一小段），可用
`regen-variants` 只重生成异常条目（全库扫描，不限于当前 `--per-domain`）：

```powershell
# 先扫描，不调 API
python -m ai_structural_holes.cli regen-variants --dry-run

# 确认后重生成（默认 min_chars=50，且低于基线 15% 长度也视为截断）
python -m ai_structural_holes.cli regen-variants --model deepseek/deepseek-v4-flash --concurrency 8
```

报告写入 `outputs/study1/variants_regen_scan.csv` / `variants_regen_report.csv`，
修复结果写回 `data/variant_articles/`。

## 10. 正式运行 Study 1

先定义模型。PowerShell 示例：

```powershell
$models = "deepseek/deepseek-chat"
```

正式命令：

```powershell
python -m ai_structural_holes.cli study1 \
  --models minimax/MiniMax-M3 \
  --gen-model deepseek/deepseek-v4-flash \
```

正式放大前，建议先把 `--per-domain` 改为 2，并添加 `--dry-run` 检查调用量。

## 11. 运行 Study 2–4

四个 Study 应分别运行和验收，不建议一次性全部启动。Study 5（真实 RAG 检索环）见第 12 节。

Study 2：

```powershell
python -m ai_structural_holes.cli study2 --models $models --per-domain 10 --seeds 3 --n-points 16 --query-source pool --distractors real --gen-route llm --gen-model deepseek/deepseek-chat --concurrency 4
```

Study 3：

```powershell
python -m ai_structural_holes.cli study3 --models $models --per-domain 10 --seeds 3 --query-source pool --distractors real --gen-route llm --gen-model deepseek/deepseek-chat --concurrency 4
```

Study 3 用来比较模型时，`$models` 至少应包含两个模型，并为它们配置对应平台的密钥。

Study 4（欺骗性检验，分两步）：

Study 4 只检验 **S1（证据坚实度）** 和 **S3（领域专业性）** 两个维度。每个问题会为这两个维度各生成三种目标版本：`none`（无该特征）、`genuine`（真实可核验）、`fake`（外观可信但不可核验）。**必须先单独生成材料并人工核验，再跑选择实验。**

**材料数量**：`问题数 × 2 维 × 3 版`。五个领域各 `--per-domain 50` 时，共 250 题 × 6 = **1500 篇**目标文章（进度条 `生成变体文章` 即为此数；`--distractors real` 时干扰文章不计入）。

**第 1 步：生成并冻结真假材料，导出审计表**

```powershell
python -m ai_structural_holes.cli study4-materials `
  --models deepseek/deepseek-chat `
  --gen-route llm `
  --gen-model deepseek/deepseek-chat `
  --query-source pool `
  --distractors real `
  --per-domain 50 `
  --concurrency 4
```

macOS/Linux 一行写法：

```bash
python -m ai_structural_holes.cli study4-materials --models deepseek/deepseek-chat --gen-route llm --gen-model deepseek/deepseek-chat --query-source pool --distractors real --per-domain 50 --concurrency 4
```

成功后会写出：

```text
outputs/study4/audit_sheet.csv
outputs/study4/audit_sheet_review.csv   # 按题目分组排序，便于逐条核验（推荐打开此文件）
```

同时目标文章冻结进 `data/variant_articles/`。`audit_sheet.csv` 每行一篇材料，含 `text` 及供人工填写的列：`verifiable`、`source_url_or_doi`、`verdict`、`reviewer`、`note`。

若已有 `audit_sheet.csv`，可单独生成分组版：

```powershell
python -m ai_structural_holes.cli audit-sheet-review
```

`audit_sheet_review.csv` 按「领域 → 题目 → S1(none/genuine/fake) → S3(none/genuine/fake)」排列，每题固定 6 行；若已运行过 `audit-genuine`，会附带 `verification_status` 等列。

**第 2 步：人工核验（必做，程序不会自动验收真伪）**

打开 `audit_sheet.csv`，逐条或抽样检查：

- **genuine**：来源能查到、数字与来源对应 → `verdict` 填通过；查不到或数字不对 → 打回重写。
- **fake**：读起来与 genuine 同样可信，但确实查不到或被篡改 → 通过；若一眼假（数字离谱、露馅表述）→ 打回重写。
- **none**：不含目标维度的证据/专业性，但仍应可读、与问题相关。

建议两人盲评并记录一致性。未通过核验的材料不要进入正式选择实验。

**第 3 步：跑选择实验**

核验通过后，用**相同**的 `--query-source`、`--per-domain`、`--gen-route`、`--gen-model` 运行 `study4`；程序会经变体库复用已冻结的同一批文本：

```powershell
python -m ai_structural_holes.cli study4 \
  --models deepseek/deepseek-v4-flash --gen-route llm \
  --gen-model deepseek/deepseek-v4-flash \
  --query-source pool --distractors real --per-domain 50 --seeds 3 --concurrency 50
```

Study 4 与 Study 1 使用**同一套题库和基线**（`--query-source pool` + 已冻结的 `data/base_articles/`），但目标文章是 none/genuine/fake 三版，**不要**用 `gen-variants` 代替 `study4-materials`。

空跑检查流程（不产生正式结论）：

```powershell
python -m ai_structural_holes.cli study4-materials --mock --per-domain 1
python -m ai_structural_holes.cli study4 --mock --per-domain 1 --seeds 1
```

这些数字只是安全的起步示例，不是功效分析结论。正式样本量应根据预期基线被选率、最小关注效应和聚类结构决定。

## 12. 运行 Study 5（真实 RAG 检索环）

Study 5 在前四个实验的“强制三选一”之外，增加**真实两阶段 RAG**：目标文先在一个冻结的真实语料库里参与检索，再在自然作答时被（或不被）引用。原理详见 [study5_真实检索环.md](study5_真实检索环.md)。

与 Study 1–4 相比，有两点必须记住：

1. **目标文直接复用 Study 1 已冻结的变体**（`data/variant_articles/`），不需要 `--gen-model`，也不会重新调用改写模型。
2. **必须先构建检索语料库**；API 开销主要在第二阶段的“读 top-k → 作答并标注引用”。

**前置条件**：

- `data/query_pool/` 已冻结（第 7 节）；
- Study 1 的 LLM 变体已生成并质检（第 9 节）；
- 被测模型的密钥已配置（第 5 节）。

**第 1 步：构建并冻结检索语料库**

从冻结题库的真实网页段落聚合去重，按领域写入 `data/rag_corpus/`：

```powershell
python -m ai_structural_holes.cli build-corpus
```

首次跑混合检索器时，建议顺便预热向量（会下载 `BAAI/bge-small-zh-v1.5`，耗时较长）：

```powershell
python -m ai_structural_holes.cli build-corpus --embed
```

macOS/Linux 一行写法：

```bash
python -m ai_structural_holes.cli build-corpus --embed
```

语料库与向量缓存冻结后，正式实验期间不要一边跑 Study 5 一边改 `data/rag_corpus/`。

**第 2 步：空跑检查流程（不产生正式结论）**

```powershell
python -m ai_structural_holes.cli study-rag --mock --retriever bm25 --per-domain 1 --top-k 5
```

**第 3 步：正式运行**

```powershell
python -m ai_structural_holes.cli study-rag `
  --models deepseek/deepseek-v4-flash `
  --per-domain 50 --seeds 3 `
  --top-k 8 --concurrency 50 `
  --retriever hybrid --alpha 0.5
```

macOS/Linux 一行写法（与 Study 1 相同，可用 `\` 续行）：

```bash
python -m ai_structural_holes.cli study-rag \
  --models deepseek/deepseek-v4-flash \
  --per-domain 50 --seeds 3 \
  --top-k 8 --concurrency 50 \
  --retriever hybrid --alpha 0.5
```

若被测模型走独立通道（如 `qwen3.6-flash`、`doubao-seed-2-0-mini-260428`），只需改 `--models`，并在 `.env` 中配置对应密钥（见第 5 节）。

正式放大前，建议先把 `--per-domain` 改为 2，并添加 `--dry-run` 检查调用量。

成功后会写出：

```text
outputs/study_rag/targets_manifest.csv      # 目标文复用清单（与模型无关）
outputs/study_rag/retrieval.csv             # 检索通道：每个目标是否进 top-k
outputs/study_rag/ate_retrieved.csv         # 检索通道 ATE（与模型无关）
outputs/study_rag/<model>/trials.csv        # 引用阶段原始记录
outputs/study_rag/<model>/ate_cite.csv      # 生成（引用）通道 ATE
outputs/study_rag/<model>/ei_leverage.csv   # 引用通道 EI
outputs/study_rag/<model>/e2e.csv           # 检索 × 引用的端到端分解
```

Study 5 把效应拆成两个通道：**检索通道**（目标是否被召回）和**生成通道**（目标已在上下文里时是否被引用）。建议先看 `targets_manifest.csv` 是否几乎全是 `reused`（不应大量 `template_fallback`），再看 `retrieval.csv` 的召回率，最后看各模型的 `ate_cite.csv` 与 `e2e.csv`。排版类特征（O1/O2/O3）在 BM25 检索上可能几乎无信号，这在两通道设计里是预期现象，不要因此误判实验失败。

## 13. 运行 Study 6（查询改写驱动的 RAG 检索）

Study 5 的第一阶段检索是**固定的、与被测模型无关**的（同一套 top-k 供所有模型共享），且采用"强制插入目标 + 位置轮换"的双通道分解。Study 6 有两点不同：一是让**被测模型参与检索决策**（检索前先由模型基于用户问题改写/扩展出若干检索式，驱动同一个 `HybridRetriever`），因此**检索结果因模型而异**；二是采用**真实端到端 RAG**——不强制插入目标、不位置轮换。

关键设计：

- **改写看不到目标文**：改写提示词只含用户问题 + 领域，绝不含候选/目标正文。于是同一模型、同一问题下，处理文和对照文面对完全相同的改写与竞争环境。
- **改写按 (模型, 问题) 复用**：一次改写用于该问题下全部 16 个目标，保证配对环境一致。
- **真实端到端**：候选集就是改写查询**实际检索到**的 top-k，按检索分数自然排序。目标只有真被检索进 top-k 才出现（在其真实 rank 位置），否则缺席、无从引用（`y=0`）。因此 `y` 直接是"目标最终是否被引用"的端到端结果，检索与生成合为一个结局。
- **取消位置轮换**：每个目标只生成 1 次（Study 5 是 `top_k` 次），生成阶段调用量约为原来的 `1/top_k`；`--top-k` 现在只决定上下文大小，不再放大调用量。

前置条件与 Study 5 相同（已冻结题库、Study 1 变体、`build-corpus` 已跑完、被测模型密钥已配置）。

**离线冒烟（无需密钥/向量模型）：**

```powershell
python -m ai_structural_holes.cli study6 --mock --retriever bm25 --per-domain 1 --top-k 5 --models mock/model-a
```

**正式运行：**
被测模型：
1. deepseek/deepseek-v4-flash
2. qwen3.6-flash
3. doubao-seed-2-0-mini-260428
4. minimax/MiniMax-M3
5. kimi/kimi-k2.5
6. openai/gpt-5.4-mini
7. google/gemini-3.1-flash-lite
8. anthropic/claude-haiku-4.5
9. meta-llama/llama-4-maverick
10. mistralai/mistral-small-2603

```bash
python -m ai_structural_holes.cli study6 \
  --models mistralai/mistral-small-2603 \
  --per-domain 50 --seeds 1 \
  --top-k 8 --concurrency 50 \
  --retriever hybrid --alpha 0.5 \
  --n-queries 3 --fuse max
```

- `--n-queries`：每个问题让模型改写出的检索式上限（默认 3）。
- `--fuse`：多检索式打分融合方式，`max`（默认，逐文档取最大分）或 `mean`（取平均）。
- 调用量：改写阶段约 `题目数 × 模型数` 次；生成阶段 `题目数 × 16 目标 × seeds × prompt × 模型数`（**不再 × top_k**）。`--per-domain 50` 单模型 `--seeds 1` 约 4000 次生成调用。

成功后会写出：

```text
outputs/study6/targets_manifest.csv           # 目标文复用清单（与模型无关）
outputs/study6/retrieval_by_model.csv         # 跨模型检索通道对比
outputs/study6/<model>/rewrites.csv           # 审计：该模型把问题改写成了什么检索式
outputs/study6/<model>/retrieval.csv          # 检索通道（参考）：该模型下每个目标是否进 top-k
outputs/study6/<model>/ate_retrieved.csv      # 检索通道 ATE（参考，按模型）
outputs/study6/<model>/trials.csv             # 生成阶段原始记录（真实 top-k）
outputs/study6/<model>/ate_e2e.csv            # 端到端被引用 ATE（本 Study 核心结局）
outputs/study6/<model>/ei_leverage.csv        # 端到端 EI
```

与 Study 5 的核心区别：一是检索通道从"与模型无关、算一次"变为"每个模型一份"；二是 Study 6 是真实端到端（`y` 已含检索+引用），不再有独立的"引用通道"与乘积式 `e2e` 分解，`ate_e2e.csv` 是核心结局，`ate_retrieved.csv` 仅作参考。设计与效度边界详见 [study6_查询改写检索.md](study6_查询改写检索.md)。

**断点续跑（`--resume`，省钱关键）：**

生成阶段调用量大，中途被限流或被内容安全拦截（如 Kimi `content_filter` 400）不该从头再来。

- **默认容错**：某次生成调用被永久拒绝（`content_filter`/`high risk`/纯 400）时不再拖垮整批，而是给该 trial 记哨兵行（`parse_ok=0` + `api_error`）后继续；瞬时错误（限流/网络/5xx）仍自动重试。被拦截的行在 `ate_e2e`/`ei` 里按**缺失数据剔除**（不计入分母），不会误当成"没引用"而低估效应。
- **续跑**：命令后加 `--resume`，会复用各模型子目录已保存的 `rewrites.csv`（不再重复调模型改写）、本地重算检索，按**语义键**（`query_id/model/prompt_style/seed/target_dim/role/pair_id`，都是 `trials.csv` 里已有的列）跳过已完成的 trial，**只补跑没跑过的生成调用**；开跑前还会自动对 `trials.csv` 去重（保留首次），修复此前中断可能留下的重复行。不加 `--resume` 则重跑即从头、清空旧 CSV。

```bash
# 第一次（中途被打断也没关系，进度已即时落盘）
python -m ai_structural_holes.cli study6 --models kimi/kimi-k2.5 --per-domain 50 --seeds 1 --top-k 8

# 断点续跑：只补缺口，阶段0≈0、检索本地免费、只在缺失的生成调用上花钱
python -m ai_structural_holes.cli study6 --models kimi/kimi-k2.5 --per-domain 50 --seeds 1 --top-k 8 --resume
```

## 14. 常用参数

| 参数 | 含义 |
| --- | --- |
| `--models` | 负责选择候选文章的模型，多个模型用逗号分隔 |
| `--per-domain` | 每个领域使用的问题数 |
| `--seeds` | 每个候选条件的重复次数 |
| `--set-size` | 每场候选文章数量，默认 3 |
| `--gen-route llm` | 使用冻结基线进行 LLM 改写 |
| `--gen-model` | 负责生成变体的模型 |
| `--query-source pool` | 使用冻结真实题库 |
| `--distractors real` | 使用真实网页段落作为竞争者 |
| `--concurrency` | 并发调用数量；限流时调小 |
| `--output-mode` | 模型输出格式：`minimal`（默认，仅 `{"choice":"A"}`，更省 token）或 `full`（含 ranking/scores/reason，便于审计） |
| `--no-progress` | 关闭进度条 |
| `--no-variant-store` | 不读写冻结变体库，正式实验一般不使用 |
| `--n-points` | Study 2 的组合数量 |
| `study4-materials` | 只生成 Study 4 的 S1/S3 真假材料并导出 `audit_sheet.csv`，不跑选择实验 |
| `build-corpus` | 从冻结题库聚合真实段落，按领域冻结 Study 5 检索语料库（可选 `--embed` 预热向量） |
| `study-rag` | Study 5：真实 RAG 检索环；复用 Study 1 冻结目标文，分检索通道与引用通道 |
| `--top-k` | Study 5/6：检索返回并作为上下文的文档数（默认 8）。Study 6 中它只决定上下文大小，不再放大调用量（无位置轮换） |
| `--retriever` | Study 5/6：检索器，`hybrid`（默认，BM25+向量）、`bm25`、`dense` |
| `--alpha` | Study 5/6：混合权重，`score = alpha·向量 + (1-alpha)·BM25`（默认 0.5） |
| `study6` | Study 6：查询改写驱动的**真实端到端** RAG（无强制插入、无位置轮换）；检索因模型而异，保留 Study 5 固定检索双通道设计作对照 |
| `--n-queries` | Study 6：每个问题让模型改写出的检索式上限（默认 3） |
| `--fuse` | Study 6：多检索式打分融合，`max`（默认）或 `mean` |

## 15. 估算调用量和费用

在 Study 命令末尾添加 `--dry-run`：

```powershell
python -m ai_structural_holes.cli study1 --models $models --per-domain 10 --seeds 3 --dry-run
```

dry-run 会估算选择阶段的调用次数和费用。需要注意：

- 价格是按 `--price-in`、`--price-out` 给出的粗略混合价格；
- 不同模型的真实价格不同；
- 估算未计缓存命中；
- 当前 dry-run 不完整覆盖基线和变体生成费用；
- dry-run 使用测试材料估算平均输入长度，正式 LLM 文章更长时实际费用可能更高。

因此它适合检查数量级，不是账单承诺。

## 16. 怎样检查结果

先看各模型子目录下的 `trials.csv`，再看统计图。路径形如 `outputs/study1/deepseek__deepseek-chat/`（`/` 在目录名中会变成 `__`）。

建议顺序：

1. 记录数是否符合预期；
2. `model`、`domain`、`target_position` 是否完整；
3. `parse_ok` 比例是否足够高且处理/对照接近；
4. `generator` 是否全部为预期路线；
5. 是否存在 `template_fallback`；
6. 再看 ATE、EI 和回归结果。

**Study 1–6 默认即时写入**：每条 trial 完成后立刻 append 到 `outputs/<study>/<model>/trials.csv`（Study 6 改写阶段同理写入 `rewrites.csv`）。运行过程中会按 `--analysis-refresh-every`（默认 100 条）或 `--analysis-refresh-sec`（默认 300 秒）重算并覆盖分析 CSV（`ate.csv`、`ei_leverage.csv` 等）；PNG 图仍只在全部跑完后生成。若中途中断，可直接从已有 `trials.csv` 手动重算分析。调试时可加 `--no-incremental-output` 回退为跑完后一次性写 CSV。

主要文件（均在 `outputs/<study>/<model>/` 下，`<model>` 为模型 slug）：

| 文件 | 用途 |
| --- | --- |
| `trials.csv` | 每次选择的原始记录（该被测模型） |
| `ate.csv` | Study 1 的效应方向、幅度和区间 |
| `ei_leverage.csv` | 特征的 EI 排名及对应 ATE（Study 1/2/3 单模型） |
| `coefficients.csv` | Study 2 的主效应和交互 |
| `ei_by_domain.csv` | Study 3 的分领域结果（单模型） |
| `deception.csv` | Study 4 的欺骗指标 |
| `delta_ei.csv` | Study 4 的真实/伪造 EI 差异 |
| `ate_retrieved.csv` | Study 5 检索通道 ATE（`outputs/study_rag/` 根目录，与模型无关）；Study 6 则在各模型子目录（检索因模型而异，作参考） |
| `ate_cite.csv` | Study 5 引用（生成）通道 ATE |
| `ate_e2e.csv` | Study 6 端到端被引用 ATE（核心结局，检索+引用合一） |
| `e2e.csv` | Study 5 端到端三通道分解（Study 6 已取消乘积式分解） |
| `rewrites.csv` | Study 6 改写审计：各模型把问题改写成了什么检索式 |
| `retrieval_by_model.csv` | Study 6 跨模型检索通道对比（`outputs/study6/` 根目录） |

Study 4 材料生成（与选择模型无关）仍在 `outputs/study4/audit_sheet.csv`。Study 5 的检索明细在 `outputs/study_rag/retrieval.csv`，目标文复用清单在 `outputs/study_rag/targets_manifest.csv`。Study 6 的检索明细按模型分别落在 `outputs/study6/<model>/retrieval.csv`。

## 17. 缓存与冻结材料

项目有三层复用：

- `.cache/llm/`：**Study 默认不再使用** LLM 响应磁盘缓存（每条 API 结果即时写入 `outputs/`）。若需旧行为，可设 `ASH_LLM_CACHE=1` 或加 `--llm-cache`。`gen-base` / `gen-variants` 等生成命令不受影响。
- `data/variant_articles/`：保存可读、可人工审查的变体正文。
- `.cache/rag_embeddings/`：Study 5/6 语料向量缓存（与 `data/rag_corpus/` 指纹绑定）。

开启 LLM 缓存时，相同请求会命中 `.cache/llm/`，但不要把缓存当作唯一的复现保证。正式实验还应记录：代码版本、命令、模型名称、基线文件、变体文件和输出文件。

不要在正式实验中途随意删除缓存、重生成基线或使用 `--force`。这些操作会改变实验材料。

## 18. 常见问题

### 找不到 API key

检查 `.env` 的变量名。DeepSeek 与 OpenRouter 使用不同密钥。

### 出现 401、余额不足或 429

- 401：密钥无效或路由错误。
- 余额不足：给对应平台充值。
- 429：降低 `--concurrency`，例如改为 2 或 1。

### 结果几乎瞬间跑完

可能因缺少密钥回退 Mock，或（若加了 `--llm-cache`）命中 LLM 磁盘缓存。Study 默认不写 `.cache/llm/`。检查终端警告和模型配置。

### 变体中出现 template_fallback

不要直接混入正式实验。检查缺失基线、空响应和生成模型配置，重新生成或剔除整对材料。

### 解析失败较多

查看 `parse_ok`，确认模型是否按 JSON 输出。解析失败应作为缺失审计，不应简单解释为目标文章未被选中。

### 结果显著但不同领域方向相反

先检查生成方式、文章长度、解析率和样本量，再讨论理论异质性。显著不等于设计有效。

## 18. 正式实验最短检查清单

运行前：

- [ ] 题库已经人工抽检并冻结。
- [ ] 基线文章没有模板回退，正文质量合格。
- [ ] 变体完成独立语义质检和配对检查（Study 1）。
- [ ] Study 4：`study4-materials` 已生成，`audit_sheet.csv` 中 genuine/fake 经人工核验通过。
- [ ] Study 5：`build-corpus` 已跑完；`targets_manifest.csv` 中目标文多为 `reused`。
- [ ] 生成模型与 `--gen-model` 一致（Study 1–4；Study 5 不需要 `--gen-model`）。
- [ ] dry-run 调用量和预算可接受。

运行后：

- [ ] `trials.csv` 行数、模型、领域和位置符合预期。
- [ ] 报告总体及分组解析率。
- [ ] 报告独立问题数，不只报告 trial 数。
- [ ] Study 1 的 EI 只使用对应 `target_dim`。
- [ ] ATE 区间按问题聚类。
- [ ] Study 4：`deception.csv` / `delta_ei.csv` 基于已核验的真假材料解释。
- [ ] Study 5：分别报告检索通道（`ate_retrieved.csv`）与引用通道（`ate_cite.csv`），并用 `e2e.csv` 核对分解。
- [ ] Study 6：以端到端被引用（`ate_e2e.csv`）为核心结局，`ate_retrieved.csv` 作参考；说明位置偏差已计入端到端效应。
- [ ] 结论经过[效度威胁清单](validity_threats.md)验收。

一句话流程：**先空跑，再准备并冻结题库和材料；先质检材料，再花钱跑选择实验；先审计原始记录，再解释 ATE 和 EI。**
